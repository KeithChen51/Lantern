01 / Positioning

平台视觉定位

灯塔平台的视觉系统独立于服务品牌新版 VI。服务品牌资产用于对外传播与品牌内容，平台系统用于产品界面、后台、知识库和工作流。

暖纸色底

页面底色采用温暖纸面色，形成内部知识平台和阅读界面的可信氛围。背景可以有轻微纹理，但不能喧宾夺主。

琥珀主色

琥珀色用于平台识别、当前导航、主操作、提示性高亮。它是经典版的识别点，不再引入探索版蓝绿色主色。

低阴影低噪声

卡片和面板以半透明白、细边框、轻阴影建立层级。避免过重玻璃、强发光、强投影和装饰性渐变。

Core / 平台核心色

纸色、墨色、琥珀、黄铜构成平台底盘。导航、标题、主按钮、页头、重点高亮只从这一层取色。

Semantic / 语义状态色

成功、警告、危险、信息优先复用黄铜、琥珀和危险色。除错误/风险外，不新增高饱和状态色。

Auxiliary / 内容辅助色

辅助色被吸收到价值标签体系，只服务内容分类和检索，不参与品牌识别、页面调性和核心操作。

吸收原则 辅助色不进入主色板，但进入设计系统的命名 token。每个价值只保留一个 base 色，再由系统生成 soft、line、text 三个派生值，避免页面里散落临时 hex。
  • --lh-value-truth保留 #6ab0a5，用于求真、事实、透明、澄清。
  • --lh-value-good保留 #728a69，用于尽善、支持、改进、协调。
  • --lh-value-order从 --lh-brass 派生，替代原紫色，用于致美、秩序、专业、工艺。
  • --lh-value-care谨慎保留 #c74f5a，用于大爱、关怀、尊严；不替代 danger。
  • --lh-value-result从 --lh-classic-amber 派生，替代原桃橙，用于幸福、结果、归处。

03 / Typography

字体与排版

经典版的文本气质来自黑体 UI 字体与少量宋体标题的组合。普通控件和表格使用黑体栈，页面主标题、价值阅读区域和少数编辑性文本可以使用衬线。

在服务现场，看见人的真实需要。 标题使用思源宋建立“手册、灯塔、价值文本”的编辑感。正文保持足够对比，不用过低透明度制造高级感。
  • UI 字体--font-hei：苹方优先，用于导航、表单、按钮、后台表格和普通 UI 正文。
  • 标题字体--font-serif：思源宋优先，用于页面主标题、价值阅读标题和编辑性段落。
  • 排版基准正文从 15-16px 起步，控件 14px，说明文字 12-13px；页面标题只在真正的顶层场景使用 40-72px。
  • 阅读行高正文 1.7-1.8，控件 1.4-1.5，标题 1.12-1.2。行距服务阅读，不为装饰拉开。

标题层级样式

标题层级不是单纯字号选择，而是页面结构。每个视图优先使用 Display、Page、Section、Subsection、Card、Kicker 六类角色，不新增临时标题样式。

04 / System Shell

桌面端与移动端系统框架

经典版桌面端参照当前产品的固定侧栏、顶部工具区和内容工作区。移动端不压缩桌面侧栏，采用底部导航作为五个核心入口的默认框架。

Desktop Shell / 当前参照

桌面端保留固定左侧导航，支持 240px 展开和 80px 折叠。主内容区根据侧栏状态偏移，顶部保留搜索、身份与关键操作，页面内容保持 1220px 最大宽度。

Mobile Shell / 目标框架

手机端主任务优先，顶部只保留页面标题、搜索或必要状态；五个核心入口进入底部导航。底部导航不承载二级菜单，复杂筛选进入页面内抽屉或分段控件。

本心 镜鉴 笃行 共创 路引
Desktop固定侧栏 + 顶部工具条 + 内容工作区。适合阅读、后台维护、问答和工作流并存的内部平台。
Tablet横屏可保留侧栏；竖屏过渡为紧凑顶部栏和底部导航，不让内容被窄侧栏挤压。
Mobile底部五入口导航，触控目标不低于 44px；主操作靠近内容或底部，避免依赖 hover。
Content不同终端共享同一信息架构，但页面骨架可以重排：桌面主从布局，移动端单列纵向阅读。

09 / Accessibility

可访问性规范

灯塔平台是内部工作系统，清晰可用优先于装饰。可访问性规则必须覆盖颜色、键盘、触控、语义和动效。

Contrast

正文、表格、按钮文字目标不低于 4.5:1；大标题、图标和边界目标不低于 3:1。低透明度文字不能承担关键说明。

Keyboard

所有可交互元素必须有可见 focus-visible，焦点顺序跟阅读顺序一致，不能只依赖 hover 暴露动作。

Touch

移动端和触控场景点击目标不小于 44x44px；底部导航、图标按钮和表单控件必须保留可触达间距。

Semantics

图标按钮必须有 accessible label；错误、帮助文本通过 aria-describedby 关联，错误字段使用 aria-invalid。

AreaRequired RuleNot Allowed
Color状态必须同时使用文字、图标或标签，不只依赖颜色。只用红/绿表达通过或失败。
Focus使用 Classic Amber focus ring，不能被阴影或透明背景吞掉。移除 outline 后不提供替代焦点。
Motion遵守 prefers-reduced-motion；动效只解释状态变化。装饰性循环动画、闪烁、强弹性动效。

交互状态矩阵

组件不能只定义默认态。按钮、输入、标签、表格行和卡片需要共享一套状态语言，让用户知道当前能否操作、正在发生什么、下一步在哪里。

StateVisual ContractBehavior Contract
Hover / Pressed改变背景、边框或轻阴影，不改变布局尺寸。只在可点击元素上出现；移动端不依赖 hover。
Focus-visible--shadow-focus + 明确边框。键盘可达，焦点顺序与 DOM/阅读顺序一致。

11 / Component Anatomy

组件结构与变体

组件规范不仅定义外观，还要定义结构、职责、变体和禁用场景。新增组件前先确认现有 primitive 是否能表达同一语义。

Button

  1. 容器：高度 32/40/48 或默认 44px。
  2. 内容：动词标签，可带 Solar 图标。
  3. 状态：hover、pressed、focus、disabled、loading。
  4. 禁用：不用按钮承载纯导航说明。

Field

  1. 可见 label。
  2. 输入控件。
  3. helper text。
  4. validation message。
  5. 错误必须和字段关联。

Chip / Badge

  1. 区分分类、状态、角色、指标、代码。
  2. 状态使用语义色。
  3. 价值标签只做内容辅助。
  4. 不要用一个 badge 表达所有含义。

Card / Panel

  1. Card 是重复对象。
  2. Panel 是工作区域。
  3. 标题、元信息、正文、操作轨道稳定。
  4. 禁止卡片套卡片制造层级。

Table

  1. 列头可排序时显示方向。
  2. 选中行显示范围。
  3. 批量操作显示数量。
  4. 空状态区分无数据和无结果。

Navigation

  1. 一级入口稳定。
  2. 深页提供返回或面包屑。
  3. 移动端五入口底部导航。
  4. 局部筛选不进入全局导航。
12 / Content & Recovery

内容、状态与恢复

Lighthouse 的文案要帮助用户判断下一步。状态信息必须说明发生了什么、影响范围、用户能做什么。

Action Copy

按钮使用具体动词：发布指南、保存草稿、重新生成、清除筛选。避免“确定”“提交”“OK”这类缺少范围的标签。

Empty State

首次空状态说明下一步；筛选空状态说明筛选条件并提供清除入口；权限空状态说明申请或返回路径。

Error Recovery

错误必须包含原因、影响和恢复动作。字段错误靠近字段，区块错误放 callout，阻塞错误提供重试或返回。

Hermit Boundary

Hermit 回答要区分直接建议、原则依据、相关案例和不确定性；证据不足时给出相近逻辑和下一步提问。

StateMessage PatternPrimary Recovery
Loading正在加载“对象/范围”，不是泛泛的 loading。骨架或局部进度，避免整页空白。
Field error指出字段问题和正确格式。聚焦字段并保留用户已输入内容。

13 / Governance

14 / Page Patterns

页面模式

灯塔平台不是每页都使用卡片墙。页面模式必须由任务关系决定，保持平台级一致性。

阅读页

标题和正文全宽，证据、目录、推荐内容使用 8/4 主从布局。正文优先 16px 阅读字号。

工作流页

表单和审核信息使用 6/6 或全宽表格。主操作固定在可预期区域，不随机漂移。

后台页

数据比较优先表格，不强行卡片化。筛选、批量操作和状态反馈保持同一骨架。

15 / Guardrails

使用边界

经典琥珀版本要有效、雅致。有效意味着规范能直接落地；雅致意味着有足够秩序、留白和克制。

Do

  • 用苹方承载产品 UI，用思源宋承载少量价值表达。
  • 用 12 列逻辑网格和固定间距控制页面秩序。
  • 用真实 Solar 图标和 lighthouseIcons 入口。
  • 用轻边框、暖纸色和低阴影表达层级。

Don't

  • 不要混入探索版蓝绿色、印刷稿或对外品牌 VI 资产。
  • 不要用字符、手绘图形或其他图标库替代 Solar。
  • 不要为了“丰富”随机跨列、叠卡、加重阴影。
  • 不要用低对比文字或过度透明制造所谓高级感。

This visual specification documents the Lighthouse platform UI system only. Service-brand VI assets should be stored and referenced separately.
