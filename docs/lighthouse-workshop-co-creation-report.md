# 精诚服务共创 Workshop 机制与共创页面说明

> 主题：从 Do / Don’t 清单到一线服务准则的长期生成机制
> 适用场景：品牌升级汇报、产品方案说明、内部共创启动沟通
> 版本：2026-05-18

## 一、为什么从 Do / Don’t 清单开始

精诚服务的“本心”回答的是品牌相信什么、坚持什么、希望给客户什么样的服务感受。但一线人员每天面对的不是抽象命题，而是具体场景。

客户等待时间超出预期时，服务顾问应该先解释进度，还是先安抚情绪？客户质疑维修费用时，怎样说明才不显得强推？旧件展示到什么程度算透明，什么表达又会制造新的不信任？雨天交车、投诉升级、保险理赔、车间延期、客户临时改约，这些场景都要求员工立刻判断。

如果“本心”不能进入这些判断，它就很难稳定发生。

我们研究胖东来岗位职责和工作流程标准时，看到的不是一份单纯的岗位说明书，而是一种服务文化落地方法。它会写清楚一个岗位应具备的能力、承担的职责、上岗前后怎么做、日常工作标准是什么，甚至突发事件发生后先做什么、再做什么、哪些事情不能做。

这给我们的启发很直接：好的服务文化不能只靠理念宣贯维持。它要继续往下拆，拆到岗位、流程、场景和动作。

Do / Don’t 清单就是这个拆解动作的第一种产品化表达。

Do 说清楚这个场景下建议怎么做。Don’t 说清楚哪些行为会伤害客户信任、服务体验或品牌立场。How 补充具体步骤，但不强迫每一条都一开始就写成完整 SOP。很多一线经验最初只是一句话、一个提醒、一个现场判断，先接住它，再慢慢整理。

## 二、胖东来研究给我们的关键启发

胖东来资料对我们有三点启发。

第一，服务理念要进入岗位职责。

胖东来的岗位职责不是停留在“负责某某工作”，而是把文化信仰、制度执行、服务标准、人才培养、安全意识、突发事件处理写进同一套职责体系里。岗位不是孤立的工作分配，而是理念进入组织运行的接口。

对精诚服务来说，这意味着 Do / Don’t 不能只写给服务顾问。维修人员、理赔顾问、备件人员、洗车人员、客休区服务人员和后台支持人员，都在不同位置影响客户体验。客户感受到的服务，不只来自前台话术，也来自车间是否准时、报价是否清楚、配件是否准确、交车是否体面。

第二，流程要从员工中来。

胖东来工作流程材料里有一个很重要的表述：流程需要每个岗位的人员积极参与，多次讨论，反复修改和实践。它不是管理者一次写完后向下发放，而是在使用中不断校正。

这正好解释了为什么我们不能只由项目组写完所有 Do / Don’t。项目组可以建立方法和框架，但不能替代一线经验。服务知识天然分布在现场。某个岗位每天反复遇到的真实问题，往往比外部观察更细，也更能暴露规则是否可用。

第三，好规则不是抽象要求，而是“场景触发 + 具体动作 + 适度裁量”。

在胖东来规则研究中，我们看到大量规则不是“要热情”“要重视客户”这样的泛化表达，而是更具体的动作。例如遇到顾客打碎物品时，先关切是否受伤，再安抚情绪，并及时清理。这样的表达降低了一线员工揣摩服务尺度的成本。

同时，胖东来并没有把员工变成只会照本宣科的人。很多规则保留了现场裁量权：什么情况需要即时处理，什么情况可以先安抚再补流程，什么情况要维护价值观底线。这一点对汽车售后尤其重要。汽车售后有高客单价、高专业壁垒、长等待周期和强信任风险，完全僵硬的规则反而可能伤害服务。

所以，我们不是照搬胖东来，而是吸收它的规则生成逻辑：价值观要落到动作，动作要来自现场，标准要保留必要的判断空间。

## 三、我们在前期编写中遇到的真实问题

项目组前期可以先写样例，也应该先写样例。没有样例，一线很难理解什么叫 Do / Don’t，也很难判断提交颗粒度。

但试写之后会发现，单靠项目组很快会遇到边界。

汽车售后岗位之间差异很大。服务顾问知道客户在前台怎么表达不满，维修人员知道车间实际为什么延期，理赔顾问知道保险沟通中哪些话容易引发误解，备件人员知道配件不到位如何影响交付，洗车人员和客休区服务人员也会看到客户情绪变化的细节。

这些经验如果不进入规则库，Do / Don’t 清单就会只像“管理者想象中的一线动作”。它可能正确，但不一定贴身；可能完整，但不一定好用。

所以，共创不是因为我们写不出来。更准确地说，是服务准则本来就应该经过一线验证。

项目组负责定义框架，品牌管理员负责审核和整理，一线人员负责带回真实场景。三者缺一块，清单都会失真。

## 四、Workshop 的定位

Workshop 不是一次线下活动，也不是一个普通意见征集入口。

它是一套长期开放的服务准则生成机制：让一线人员围绕真实岗位和真实场景提交经验，由系统先做基础质检，再交给品牌管理员审核、编辑和发布，最后形成全员可查阅、可学习、可复用的 Do / Don’t 指南。

它的逻辑链路是：

```text
胖东来研究
-> 形成“理念要转成岗位动作”的方法判断
-> 项目组先写样例，搭建 Do / Don’t 结构
-> 一线围绕真实岗位和真实场景持续提交
-> 系统做基础质检，减少重复和明显不完整内容
-> 品牌管理员审核、编辑、归类、发布
-> 已发布内容进入公共指南、培训材料和 Hermit 知识来源
```

Workshop 首期不做大而全。它不做泛论坛，不做复杂投稿社区，也不做多层级官僚审批。首期只聚焦一件事：围绕汽车售后一线岗位，持续生成高质量 Do / Don’t 服务动作卡。

这个边界越清楚，机制越容易跑起来。

## 五、共创内容的结构设计

共创提交不能要求一线员工“写文章”。一线最熟悉的是现场，不一定熟悉品牌语言。如果表单一上来就要求写完整案例、价值观解释和标准化表述，很多真实经验会被挡在外面。

更合适的方式，是让他们填写一张“服务动作卡”。

一张服务动作卡建议包含这些字段：

| 字段 | 用途 |
| --- | --- |
| 标题 | 用一句话概括这条经验 |
| 岗位 | 标明适用于服务顾问、理赔顾问、维修人员、备件人员、洗车人员、客休区服务人员或其他支持岗位 |
| 服务场景 | 标明发生在等待超时、费用说明、故障解释、交车提醒、投诉安抚、雨天接送、旧件展示等场景 |
| 要解决的问题 | 用一两句话说明现场痛点 |
| Do | 这个场景下建议怎么做 |
| Don’t | 这个场景下不要怎么做 |
| How | 如果有更具体步骤，可以补充 |
| 关联本心 | 可选；一线不确定也可以留给管理员补充 |
| 门店和贡献人 | 用于后续展示和贡献记录 |

这里有一个重要原则：Do 和 Don’t 可以只填其中一个，也可以都填。

因为真实经验不总是成对出现。一线有时先意识到“这样做有效”，有时先意识到“这个做法千万不要再出现”。如果强行要求 Do 和 Don’t 必须同时填写，反而会增加编造和凑字的可能。

系统应该先接住真实经验，再通过审核和编辑把它整理成更完整的服务指南。

## 六、初审机制：把好内容入口

Workshop 首期设置“自动初筛 + 管理员审核”的两段式审核机制。

自动初筛承担的是内容入口管理，不承担最终价值判断。它主要解决两个基础问题：一是提交内容是否成形，二是是否与已发布内容明显重复。

从内容完整度看，系统会检查标题、岗位、场景、Do 或 Don’t 是否具备基本信息，避免大量空泛内容直接进入审核队列。从重复性看，系统会结合标题、岗位和服务场景，识别是否已经存在高度相似的发布内容，减少重复整理成本。

这一阶段的 AI 能力定位为“基础质检”，而不是“AI 审稿”。它不判断内容是否代表品牌立场，也不直接决定发布。通过初筛的内容进入品牌管理员审核；未通过的内容回到提交者个人区，并给出补充方向。

随着数据积累，初筛能力可以逐步扩展到结构完整性、动作可执行性、场景相关性、风险识别、表达清晰度和本心原则关联。但无论能力如何演进，最终发布权仍然保留在品牌管理员手中。

## 七、审核机制：把一线经验整理为品牌准则

品牌管理员在 Workshop 中承担“经验整理者”和“准则发布者”的角色。

一线提交的内容往往带有现场语言。它可能很短，也可能只包含一个提醒、一个禁忌或一个处理动作。这类内容的价值在于真实，但还不能直接作为品牌准则发布。管理员需要在审核中判断三件事：这条经验是否真实存在，是否具备推广意义，是否符合“本心”原则和品牌表达。

通过审核的内容会进入编辑和归类环节。管理员可以补充场景描述，统一措辞，合并重复经验，标注适用岗位和服务场景，再发布到公共可见区。被退回的内容则保留修改入口，让提交者根据反馈继续补充。

这套机制避免了两个极端：一端是完全由项目组编写，容易脱离现场；另一端是完全开放投稿，容易变成松散内容流。Workshop 选择的是中间路径：一线贡献真实经验，品牌管理员完成准则化整理。

首期后台角色保持简单。品牌侧设置一个最高管理员角色，负责审核、编辑、发布和维护公共指南。审核功能归属于共创模块内部，不单独成为 Lighthouse 的第六个产品板块。对员工来说，平台仍然保持五个入口：本心、镜鉴、笃行、共创、路引。

## 八、共创页面：数字化承载与运行闭环

共创页面的作用，是把一套原本容易停留在线下讨论、微信群收集或表格汇总中的机制，转化为可持续运行的产品流程。

没有数字化承载时，共创内容往往会出现几个问题：格式不统一，修改过程不可追踪，审核状态不透明，优秀经验难以展示，也难以进入后续培训和知识库。共创页面要解决的不是“展示一个表单”，而是把提交、初筛、修改、审核、发布、学习和复用连成闭环。

页面结构可以分为三个区域：

- 公共可见区承载已发布成果。用户进入这里看到的是经过审核的 Do / Don’t 指南，而不是讨论流。内容可以按岗位、服务场景、本心原则和关键词查询。每条发布内容保留适用岗位、服务场景、Do、Don’t、How、贡献门店和贡献人。贡献榜以“门店 + 个人姓名”呈现，让一线经验在组织内被看见。
- 提交区承载服务动作卡。它用固定字段约束内容结构，用选择项和短句提示降低表达门槛。一线员工不需要写完整文章，只需要围绕岗位、场景、问题、建议动作和禁忌动作提交经验。Do 和 Don’t 可以单独提交，也可以同时提交，系统先接住真实经验，再通过后续审核整理为完整指南。
- 个人区承载个人提交记录和反馈闭环。提交者可以看到草稿、待审核内容、退回原因、已发布内容和历史记录。内容被退回时，页面反馈不以“否定”为核心，而是指出需要补充的具体方向，使提交者知道如何继续完善。

在运行上，Workshop 不做一次性活动，而是作为长期开放的共创机制嵌入 Lighthouse。可以先按几个阶段推进：

- 启动阶段，由项目组提供少量种子样例，覆盖客户等待超时、费用说明、旧件展示、维修延期、投诉安抚、雨天交车等高频场景，用来说明一条合格 Do / Don’t 的颗粒度。
- 开放提交后，服务顾问、理赔顾问、维修人员、备件人员、洗车人员、客休区服务人员和其他后台支持人员都可以参与。客户体验并不只发生在前台，车间履约、配件准确性、交车细节和后台支持同样会影响客户判断。
- 审核节奏可以按周运行。管理员集中处理初筛通过的内容，对具备发布价值的经验进行编辑和发布，对表达不完整但有价值的内容退回补充，对重复内容进行合并。前期重点不在数量，而在形成可执行的质量标准。
- 发布后的内容进入公共指南，也可以用于新员工培训、门店服务复盘、优秀经验分享和 Hermit 推荐。随着现场问题变化，已发布内容还需要定期复盘。Workshop 最终维护的不是一本静态手册，而是一套持续更新的服务动作库。

从组织价值看，共创页面改变的是服务准则的生成方式。过去，服务规范往往由管理层或项目组向下编写；Workshop 引入一线经验回流，让服务准则从真实场景中生长出来，再经过品牌审核形成统一表达。

这套机制的价值可以分开看：

- 对一线来说，现场经验被看见，好的处理动作有机会变成组织共同使用的指南。
- 对品牌管理来说，内容来源更接近真实服务现场，审核工作也有固定入口和状态记录。
- 对服务管理来说，抽象评价可以转化为具体动作讨论，复盘时不只停留在“态度好不好”，而是讨论应该怎么做、不能怎么做。
- 对 Lighthouse 来说，共创补上了从理念学习到一线执行之间的中间层，使“本心”不只停留在学习内容里，也能落到岗位动作和服务场景里。

首期仍需保持清晰边界：

- 内容上，聚焦 Do / Don’t 服务动作卡，不扩展成泛论坛或开放案例社区。
- 角色上，只保留品牌侧最高管理员审核，不引入复杂权限体系。
- AI 上，只做基础质检和重复识别，不承担品牌立场判断。
- 方法上，借鉴胖东来的服务文化落地逻辑，但不逐条照搬。
- 表达上，服务于一线经验提交，不把一线员工变成材料撰写者。

推进节奏可以保持轻量。先完成机制设计和样例准备，再通过 Railway 或内部测试环境跑通提交、初筛、修改、审核和发布流程；随后选择少量岗位和门店进行内部试用，观察表单是否好填、退回原因是否清楚、管理员审核负担是否可控；机制稳定后接入内部 IAM 或企微，最后再把已发布的 Do / Don’t 指南接入 Hermit，让共创内容在问答和学习场景中被再次调用。

## 九、结论

Workshop 是精诚服务把“本心”转化为一线服务动作的长期共创机制。

它的核心不是“收集意见”，而是建立一条服务准则生成链路：一线提交真实经验，系统完成基础质检，品牌管理员审核整理，优秀内容发布为公共指南，并进一步进入培训、复盘和智能助手知识体系。

共创页面是这套机制的数字化载体。它把原本分散在门店现场、线下讨论和个人经验里的服务判断，转化为可提交、可审核、可修改、可发布、可复用的品牌资产。

从胖东来研究得到的启发，也在这里形成了精诚服务自己的落地路径：服务文化要真正进入组织，必须从理念走到岗位，从岗位走到场景，从场景走到动作。Do / Don’t 清单是第一步，Workshop 则让这一步可以长期持续。
